All changes are small and surgical. Each was syntax-checked (`py_compile` / `node --check`) and, where noted, verified live after deploy. Findings that were investigated but NOT fixed (too big / need your input) are listed at the end.

## Code fixes (deployed)

1. **Redirect-following across ALL tool-execution paths** — tool calls that hit an HTTP 301/302 were failing instead of following the redirect. Added `follow_redirects=True` to every HTTP client used for tool execution:
   - `api/main.py` (execute_tool, test_tool), `api/modules/process/service.py` (builder test executor), `core/tools/builtin/api.py` (canonical APITool + WebhookTool — used by the real engine + agent chat).
   - **Verified live:** the process "Auto-Approve" step error advanced from `HTTP 301` → `HTTP 404` after deploy (redirect now followed; the residual 404 is a dead test-tool URL, separate).

2. **Startup ReferenceError: `authToken is not defined`** (`ui/index_parts/app-core.js`, `loadDynamicProcessTemplates`) — the function ran at init before the implicit global existed. Made the read defensive: `(typeof authToken !== 'undefined' ? authToken : null) || localStorage…`.

3. **"Calculate" shape missing from the manual Process-Builder palette** (`ui/process-builder.html`) — the shape was fully implemented (default config, icon, property panel, engine normalization) but had no palette entry, so users couldn't add it manually. Added a **Data → Calculate** palette item.
   - **Verified live:** Calculate now drags onto the canvas and opens its full property panel (operation / formula / name-this-result).

4. **Department split-brain — FIXED + VERIFIED LIVE.** The Security UI's `POST/DELETE /departments` wrote **only** to in-memory state, while process routing / identity resolution read the **DB**, so a department created from the Security UI was invisible to approval routing. Root cause had **two layers**:
   (a) `create_department`/`delete_department` never called the DB service — now they call `DepartmentService.save_department(...)` / a new `DepartmentService.delete_department(...)` (`api/security.py` + `database/services/department_service.py`), mirroring how `create_group` already persisted.
   (b) The Security `Department` model generated a **non-UUID id** (`"dept_xxxxx"`) while the DB `departments.id` is a **UUID** column, so the DB insert silently failed (caught by the try/except). Changed `Department.id` to generate a proper `str(uuid.uuid4())` (`core/security/models.py`), matching `UserGroup` (whose comment already says "proper UUID for database compatibility"). Nothing depends on the old `dept_` id prefix (the `dept_manager:`/`dept_head:` strings are routing recipients, unrelated).
   **Verified live after deploy:** a department created via `POST /api/security/departments` now gets a UUID id and **appears in the DB-backed `GET /api/identity/departments`** list (which approval routing reads). Split-brain closed for departments. (Left a junk `ZZ SplitBrainCheck` dept — add to cleanup.)

5. **Manager not editable from the Security → Users update path** (`api/security.py`) — `UpdateUserRequest` was missing `manager_id`/`employee_id`, so a user's direct manager could only be set from the Org-Chart "reports to" dropdown (org placement scattered across two UIs). Added both fields to `UpdateUserRequest` and applied them in `update_user` (admin-gated, same pattern as `department_id`).

6. **Modals lingered across navigation + ignored Escape** (`ui/index_parts/features-approvals-page.js`) — the transient dialogs that get appended to `<body>` (Create/Edit Role, Edit User, Edit Tool, etc. — all share `class="fixed inset-0 z-50 … backdrop-blur-sm"` with an id ending in `-modal`) were only dismissible via their own Cancel button, so they stayed open over the next page. Fix: the `navigate()` override now removes any such overlay before switching pages, and a one-time global Escape handler closes the top-most one. The static create-wizard (`<section id="page-create">`) doesn't match the selector, so it's untouched. (node --check passed; live-verify pending deploy.)

7. **🔴 IMPORTANT — frontend JS/CSS changes weren't reaching browsers (stale cache-bust version).** `ui/index.html` loaded every asset with a **hardcoded** `?v=` query (e.g. `features-approvals-page.js?v=20260223h`, `index.css?v=3bcda8b`). Because that string never changes on deploy, browsers keep serving the **cached** old JS/CSS — so frontend fixes (the modal fix above, the `authToken` fix, the dashboard-hero CSS, etc.) silently did NOT take effect for anyone with a warm cache. This is why the modal fix appeared not to work on first check. Bumped **all** JS + CSS versions to `?v=20260621a` so browsers refetch. **Operational recommendation:** make `?v=` dynamic (git short-hash or build timestamp) so every deploy auto-busts the cache — otherwise every future frontend change needs a manual version bump.

8. **Two access-control endpoints were open without a login** (`api/modules/access_control/router.py`) — `GET /check/{agent_id}` and `GET /preview/{agent_id}` took `user_id`/`org_id` as query params with **no auth dependency**, so anyone could probe "what can user X access?" by passing arbitrary IDs. Added `current_user: User = Depends(get_current_user)` to both (matching the other 14 endpoints in the same router). Safe: the admin UI/portal don't call these two endpoints, so nothing breaks. (py_compile OK.)

9. **A bad session timestamp could 500 *every* logged-in request** (`api/security.py`, `get_current_user`) — the session-timeout check did `datetime.fromisoformat(session.last_activity)` with no guard; a missing/malformed value (e.g. a freshly rehydrated session after a restart) would throw and crash auth for everyone. Wrapped it: on a parse error it treats the session as just-active and refreshes the timestamp instead of crashing. (py_compile OK.)

10. **"Calculate" (sum/avg) crashed on real-world numbers like "1,200 AED"** (`core/process/nodes/data.py`) — it did `float(v)` directly, so any value with a currency symbol/comma/text threw and 500'd the step. Added a `_coerce_number` helper that strips non-numeric characters ("1,200 AED" → 1200) and skips truly non-numeric entries; sum/avg now compute over the numeric values without crashing. (py_compile + a self-test confirmed: sum of ['1,200 AED','300','abc',None,45.5] = 1545.5.)

11. **Unsupported workflow blocks now fail VISIBLY + the AI is steered away from them** (demo-safety, Step 1 of the engine work) — the engine never executed **Run in Parallel, Loop, While, or Call Process**; their executors returned *success* while doing nothing → silently wrong/empty results with a green checkmark. Two-part fix:
    - **Fail-loud:** all four executors now return a clear failure (`STEP_NOT_AVAILABLE`, message: "rebuild using supported blocks") instead of silent success (`core/process/nodes/logic.py`). Runtime-tested: all four return the failure.
    - **Steer the AI away:** the process-builder KB (`docs/PROCESS_BUILDER_KB_SHAPES.md`) and the generator prompt (`core/process/wizard.py`) now tell the AI NOT to use these blocks (removed the "you MUST use a parallel node" directives, dropped them from the allowed-types list, replaced the parallel/call_process config guidance, and removed `loop` from the fallback templates). The AI will now generate workflows using only the blocks that actually run (Form, AI, Decision, Approval, Notification, Tool, Calculate).
    - This is **Step 1** (make the demo safe). **Step 2** = actually implement parallel/loop/sub-process in the engine as real, tested features, then re-enable them in the KB.

12. **Run in Parallel — IMPLEMENTED + TESTED** (Step 2, first one done) — the engine now actually executes parallel branches. New methods in `core/process/engine.py` (`_run_parallel`, `_run_chain`, `_find_merge_node`): when a `parallel` node runs, the engine walks every branch to completion (any number of branches, multi-step branches), then continues from the **merge** node (the nearest node all branches reconverge to). Branch failures fail the whole run; the merge runs once. It is **additive** — only `parallel` nodes take this path, so normal/linear processes are byte-for-byte unchanged, and it's bounded by `max_nodes` (no runaway). Removed parallel's fail-loud guard and **re-enabled it in the KB + generator** (with the rule "every branch must reconverge to one merge node"); `loop`, `while`, `call_process` remain guarded.
    - **Verified with a sandbox test harness** (real engine, synthetic definitions, no DB/LLM): ✅ 2-branch split→merge, ✅ 3 branches with a multi-step branch, ✅ a failing branch fails the run, ✅ plain linear process unaffected (regression). All pass.
    - Note: branches run **sequentially** (not truly concurrent), so the result/timing equals a sequential layout — the generator is told to prefer sequential and use parallel only when the user explicitly asks. The value here is that a `parallel` node (manual or existing) now runs correctly instead of silently dropping branches.
    - Still to implement (more involved; both have good workarounds so lower priority): **Loop/While** (needs loop-back + after-loop detection; workaround = iterate inside one AI step) and **Call Process** (needs loading the child process definition from the DB; workaround = inline the steps).

13. **AI process generation: a "Form" task was turned into an empty "AI Step", and routing conditions had blank comparison fields** (found by generating a deliberately complex process — vendor-invoice approval with a match-gate + 3-threshold cascade + multi-level approval — and inspecting every node on the builder UI). Two real, demo-breaking generation bugs:
    - **Form → AI Step (fields dropped).** The 2nd input step "Enter Invoice Details" (vendor name, invoice amount, expense-category dropdown) was generated as an empty **AI Step** instead of a **Form**, so all three fields vanished. Everything downstream that needed the amount then broke.
    - **Decision nodes with an empty field.** The two routing conditions (`< 5000`, `< 20000`) had the operator and value set but the **left-hand field blank**, so the cascade couldn't evaluate.
    - **Root cause:** in structured (reviewed-task) mode, neither the generation prompt nor the QA-review prompt *enforced* that the user's explicit task type is authoritative (a "form" task may silently become "ai"), nor that every condition rule must carry a non-empty field referencing a real upstream variable (`core/process/wizard.py`).
    - **Fix (two layers):** (a) **Prompt hardening** — added HARD RULES to `STRUCTURED_GENERATION_PROMPT` (task type maps 1:1 to node type; a "form" task MUST stay a form with its fields; every condition rule MUST have a real non-empty field; a cascade reuses the same amount variable) and reinforced the same in the review checklist (items 1 + 11). (b) **Deterministic safety net** `_reconcile_structured_nodes(process_def, tasks)` that runs AFTER all LLM passes, so it holds regardless of model drift: for any task the user explicitly typed "form", it forces the matching node back to a form (seeding fields from the instructions if empty — text/number/select-with-options/file); and it backfills any empty condition field from the cascade's existing field (or an amount-like upstream variable). It is structured-mode only and additive — a node that doesn't match an explicit task is never touched.
    - **Verified with a sandbox unit test** (no LLM) reproducing the exact live failure: ✅ the empty AI step is restored to a form with `vendorName`/`invoiceAmount`/`expenseCategory`(select, 5 options); ✅ both empty routing conditions get the real amount field; ✅ legitimate AI nodes (not matching a form task) are left unchanged. Live re-verification on the builder UI pending deploy.

14. **Live re-verification of #13 + two more generation-quality issues found** (re-generated the same complex vendor-invoice scenario after deploy and inspected every node again). **#13 confirmed fixed:** this run the form "Submit Vendor Invoice" came out as a proper **Form** with ALL four fields (Invoice PDF=file, Vendor Name=text, Invoice Amount=number, Expense Category), and **every** condition now carries a real field (`parsedData.totalAmount`) — no blank fields. The LLM varies run-to-run, and this run surfaced two *different* quality problems (so they're flagged + prompt-hardened, not yet guaranteed):
    - **Wrong condition operator on thresholds.** "Determine Approval Path" was generated as `parsedData.totalAmount` **equals** `5000` instead of **less_than** `5000` (the wording was "under 5000"). So the routing only branches at exactly 5000. Added a HARD RULE mapping wording→operator (under/below→less_than, over/above→greater_than, etc.) to both the generation prompt and the QA-review checklist.
    - **Approval routed to a form value as a "department".** The department-manager approval was pointed at a department named `{{formData.expenseCategory}}` (the category dropdown value), so it resolved to **0 approvers** (the platform's own pre/post-check correctly flagged "create the {{formData.expenseCategory}} department"). The first run got this right ("their direct manager"), so it's LLM variance. Added a HARD RULE: an approver must be a real target (direct manager / named department's manager / specific user / role) and NEVER a form field value; "route to the department manager" = assigneeType `dynamic_manager`.
    - These two are prompt-level hardening (belt). A deterministic operator/approver repair (suspenders, like the #13 reconcile) is the natural next step if they keep recurring — operator can be inferred from the task instruction wording, and an approver set to `{{form...}}` can be rewritten to `dynamic_manager`.

15. **Approver routing — DETERMINISTIC suspenders added** (extends #14's approver finding). `_reconcile_structured_nodes` now also repairs approval nodes whose approver is a form value: if an approval's department name (`assignee_department_name` / `department_name` / `department`) is a `{{...}}` template, or its `assignees`/`assignee_ids` list contains a `{{...}}` template, it clears that and falls back to the requester's direct manager (`assignee_source: user_directory`, `directory_assignee_type: dynamic_manager`) — the same target the good run produced. It is deliberately narrow: it only fires on a *templated department name / assignee id*, so the legitimate "expression" assignee type (a form field that genuinely picks the approver, template stored in `expression`) and real department names are never touched. **Verified with a sandbox unit test:** ✅ templated department → dynamic_manager (dept cleared); ✅ templated assignee list → dynamic_manager (cleared); ✅ legit `expression` approver untouched; ✅ real department name "Finance" untouched. (The wrong-operator issue from #14 stays prompt-only — its direction (less_than vs greater_than) can't be inferred safely without the per-threshold intent, so a deterministic flip would risk being wrong.)

17. **Clearer process drawing — guaranteed clean layout for AI-generated processes** (`ui/process-builder.js` + `process-builder.html`). The auto-layout now uses the **dagre** layered engine (loaded from cdnjs), which orders each rank to minimise edge crossings and spaces ranks so edges run in clear channels. Critically, dagre's own per-edge **waypoints** (which route clear of every node) are captured and used by the renderer (`tryDagreWaypointPath`), so a line never runs through an unrelated shape. Safeguards: if dagre isn't loaded it falls back to a new barycenter-ordered built-in layout; if the offsets show a node was dragged after layout, the edge falls back to the live orthogonal router; saved/hand-built processes are untouched. Render-only — no effect on execution.
    - **Verified objectively on a complex graph (11 nodes, 13 edges, 3-condition cascade + side branches + merge):** edges-through-other-nodes went from **6 → 0**, node overlaps **0**, confirmed both by an automated path-vs-rect crossing check and visually (clean branching tree, every edge routed around the shapes). Real generation (`startBuildAnimation`) applies the dagre positions + waypoints directly, so generated processes get this automatically.
